## New findings (only things introduced by iter-2)

- [F-CTO-NEW-001] Vitest default pool (`threads`) flakes hard on the new PGLite recall tests. Running `npx vitest run` (no flags) in `packages/core` produces 12 failures — all timeouts on PGLite-touching suites: `default-backend`, `doctor-dim-mismatch`, `pglite-adapter`, `pglite-recall-wiring` (3 cases), `pglite-reembed-atomic`, `pglite-vector-literal` (2 cases), `reembed-migration` (3 cases), `yaml-truth-traceability` (Test B parameterized + adversarial). `npx vitest run --pool=forks` passes 855/855. PGLite WASM is process-global and does not tolerate the threaded shared-state pool when auto-embed kicks the embedder pipeline. The PR description acknowledged "1 flake" but the actual count under defaults is 12 — significant developer-experience regression and a CI risk. Fix: set `pool: 'forks'` in `packages/core/vitest.config.ts` (or scope the override to PGLite suites). Filed as iter-3 must-fix-before-merge-to-main.

- [F-CTO-NEW-002] `_autoEmbedNewEngrams` runs an O(N) `hasEmbedding` probe across **all active engrams** on every write (`packages/core/src/index.ts:2082-2087`). For a 10k-engram store, every `learn()` triggers 10k `SELECT 1 FROM engram_embeddings WHERE engram_id = ?` queries before finding the one new ID to embed. The intended cheap path was "only embed engrams missing from the table" but the loop iterates the full active set rather than asking the DB for the missing IDs in one query. Fix: replace the loop with `SELECT engram_id FROM engram_embeddings` (or `LEFT JOIN ... WHERE engram_embeddings.engram_id IS NULL`) and diff in memory. Acceptable for Sprint 0 N=hundreds, not acceptable when stores grow. Flag for iter-4 or Sprint 1.

- [F-CTO-NEW-003] `_autoEmbedNewEngrams` reads `getVectorColumnDim()` and `getEmbedder(resolveEmbedderName())` on every write (`index.ts:2071-2078`). Both are bounded but not free; on every learn this adds two DB roundtrips + an embedder factory call. Cache `indexedDim` on the adapter and `activeAdapter.dim` at constructor time (both are stable for the process lifetime once decided). Minor.

- [F-CTO-NEW-004] `recallSemantic` cold-start fallback at `index.ts:1220-1224` calls `embeddingSearch(filtered, query, limit, this.paths.root)` — which writes to and reads from `.embeddings-cache.json`. With PGLite as the active backend (M-3 default), the JSON cache is now a parallel persistent vector store on disk too, owned by no one in particular. On `_loadAllEngrams`-sized stores the cache grows alongside the pgvector column and the two will drift on embedder switches because the PGLite path is the authoritative migration target but the JSON cache is also stamped + invalidated separately. Recommend: when PGLite is active, the fallback should not write to the JSON cache (it's a transient bridge until embeddings catch up). Minor data-hygiene issue.

- [F-CTO-NEW-005] `rebuildJsonCache` (`packages/core/src/embeddings.ts:412-443`) returns `{ reembedded, skipped, reason }` but the sync({reembed}) wiring at `index.ts:2281-2288` swallows the return value via `void` IIFE. Same root cause as the punted M-11 (#272); iter-2 left the non-PGLite branch with the same silent-failure pattern it was created to fix. Same severity; same fix; flag for iter-4.

- [F-CTO-NEW-006] The adversarial Test B variant (M-2) inserts only into `engram_embeddings` table, not into the relational `engrams` table. It correctly tests the intersect-with-filtered defense (which is the wired-in defense for the public recall path) but does not test what would happen if a future change put synthetic data into the `engrams` table itself. A stronger variant would insert into both tables and assert `list()` / `getById()` still hide it. The current test is adequate but the gap should be acknowledged. Minor.
